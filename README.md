# Agentic Shell

자연어로 받은 요청을 Python 도구를 그때그때 만들어 실행하며 처리하는 작은 CLI 에이전트입니다. 만든 도구가 나중에도 쓸모가 있겠다 싶으면, 사용자에게 확인을 받고 로컬 `skills/` 디렉토리에 저장해 둡니다. 이후 세션에서 에이전트가 알아서 찾아 씁니다.

전체 제어 흐름은 표준 라이브러리 Python만으로 직접 짰습니다 — 에이전트 라이브러리도, SDK도 쓰지 않고, Groq의 OpenAI 호환 Chat Completions API를 `urllib`로 호출하는 게 전부입니다.

---

## 1. 실행 방법

### 환경 설정

Python 3.10 이상과 Groq API 키만 있으면 됩니다. 무료 티어로 충분합니다.

```bash
export GROQ_API_KEY=gsk_...
```

### LLM 설정

모델과 토큰 한도는 환경 변수로 조절합니다. 아래 조합이 무료 티어에서의 권장 시작점입니다.

```bash
export GROQ_MODEL="openai/gpt-oss-20b"   # 무료 티어에서 tool calling이 안정적
export GROQ_MAX_TOKENS=1024              # 무료 티어 TPM 한도 부담을 줄여 줌
```

모델 선택 시 알아둘 점이 몇 가지 있습니다.

- `llama-3.3-70b-versatile`(원래 기본값)은 가끔 정상적인 `tool_calls` 필드 대신 Llama 고유의 평문 형식(`<function=name{json}</function>`)으로 도구를 호출합니다. 이러면 Groq 서버 파서가 `tool_use_failed`로 400 에러를 내므로, 무료 티어에서는 `openai/gpt-oss-20b`를 권장합니다.
- `llama-3.1-8b-instant`는 TPM 한도가 넉넉하지만 에이전트 루프를 돌리기엔 역부족입니다 — 인라인 데이터를 무시하고, 같은 질문을 반복하며, 진전 없이 토큰만 소비합니다. 간단한 일회성 작업에만 쓰세요.
- 무료 티어에서는 TPM이 가장 큰 제약입니다. 매 호출이 전체 히스토리 + 도구 스키마 + 시스템 프롬프트를 보내고, 응답용으로 `GROQ_MAX_TOKENS=4096`을 예약하면 단일 요청이 분당 8k 예산의 절반 이상을 잡아먹을 수 있습니다. `GROQ_MAX_TOKENS`를 ~1024로 낮추는 게 가장 쉬운 해결책입니다.

그 외에 히스토리 보관 수를 조절하는 `GROQ_HISTORY_CAP` (기본값 20)도 있습니다.

### 실행

```bash
python3 agent.py
```

여러 줄 붙여넣기를 지원합니다 — 프롬프트와 데이터를 한꺼번에 붙여넣으면 하나의 메시지로 처리됩니다:

```
you> 다음 JSON에서 HP가 100보다 큰 몬스터 이름을 뽑고, 그들의 평균 HP를 알려줘.
[
  {"name": "Slime",  "hp":  30},
  {"name": "Orc",    "hp": 120},
  {"name": "Dragon", "hp": 800},
  {"name": "Lich",   "hp": 250}
]

agent> Plan: parse the JSON, filter hp > 100, average those. Criterion: print
       qualifying names and their mean hp.

[tool: run_python]
agent> Orc, Dragon, Lich — 평균 HP 390.0.
       이 로직을 'monster_hp_summary' 스킬로 저장해둘까요?

you> y
[saves to skills/monster_hp_summary/]
```

### 프로젝트 구조

```
agent.py        메인 루프 — API를 돌리는 while 루프 하나
tools.py        내장 도구 스키마 + 디스패치
prompts.py      4가지 운영 원칙이 담긴 시스템 프롬프트
skills/         사용자 확인을 거쳐 저장된 도구들 (스킬마다 SKILL.md + tool.py)
workspace/      run_python이 쓰는 임시 디렉토리
```

### 내장 도구

| 도구          | 하는 일                                                       |
| ------------- | ------------------------------------------------------------- |
| `run_python`  | 격리된 서브프로세스에서 Python 스크립트 실행                  |
| `read_file`   | UTF-8 텍스트 파일 읽기                                        |
| `write_file`  | workspace에 파일 쓰기                                         |
| `ask_user`    | 확인이 필요할 때 사용자에게 질문 (HITL)                       |
| `save_skill`  | 만든 도구 저장하기 — 반드시 y/n 확인을 거침                   |
| `load_skill`  | 저장된 스킬의 소스 코드 불러오기                              |
| `list_skills` | 저장된 스킬 목록과 "언제 쓰면 좋은지" 설명 보기              |

## 2. 설계 결정 사항

### SDK도, 프레임워크도 없이 직접 HTTP 호출

`agent.py`는 `urllib`로 `https://api.groq.com/openai/v1/chat/completions`에 직접 POST합니다. 제어 흐름은 `run_turn()` 안의 `while` 루프 하나가 모델을 움직이는 구조입니다: 메시지를 보내고 → `tool_calls`를 받고 → 도구를 실행하고 → `tool` role 결과를 붙이고 → 다시 반복. 의도적인 선택입니다 — 에이전트 추상화 계층을 쓰면 이 연습에서 직접 다뤄 보고 싶은 부분이 가려지니까요.

### 서브프로세스로 격리 실행

`run_python`은 모델이 작성한 코드를 `workspace/_run.py`에 쓰고, 새 `python3` 서브프로세스에서 타임아웃을 걸어 실행합니다. 가장 단순하면서도 돌아가는 격리 방식입니다 — 호출 사이에 프로세스 상태가 섞이지 않고, stdout/stderr가 깔끔하게 분리되며, 모델은 실제 traceback을 `tool_result`로 받아볼 수 있습니다.

### Traceback 피드백으로 스스로 고치기

사용자 턴 하나당 모델이 최대 12번까지 반복할 수 있습니다. `run_python`이 0이 아닌 종료 코드를 내면, 전체 STDOUT/STDERR/종료코드가 다음 `tool_result`로 돌아갑니다. 시스템 프롬프트는 모델에게 traceback을 잘 읽고 진짜 원인을 고쳐서 다시 돌리라고 지시합니다 — 대충 때려맞추지 말고요.

### 스킬은 코드가 아니라 데이터로 취급

저장된 스킬은 파일 두 개짜리 디렉토리입니다:

- `SKILL.md` — "언제 쓰면 좋은지" 한 단락 설명
- `tool.py` — 실제 코드

시작할 때 에이전트는 모든 `SKILL.md`를 읽되 설명만 시스템 프롬프트에 넣습니다(소스는 넣지 않음). 전체 소스는 에이전트가 그 스킬을 쓰겠다고 판단했을 때 `load_skill`로 불러옵니다. 스킬이 많아져도 프롬프트가 작게 유지되는 구조입니다 — 메뉴를 보고, 필요한 레시피만 꺼내 쓰는 셈이죠.

### 명시적으로 사람이 개입하는 접점 3곳

1. 요청이 모호하거나 빠진 정보가 있으면 `ask_user`로 물어봄.
2. `save_skill`은 항상 미리보기를 보여주고 y/n 확인을 받음.
3. 사용자가 매 작업을 `you>` 프롬프트에서 직접 입력.

시스템 프롬프트에서 모델에게 "추측하지 말고 `ask_user`로 확인하라"고 명시하고 있습니다.

### 시스템 프롬프트의 4가지 운영 원칙

1. 코드를 작성하기 전에 가정과 성공 기준을 먼저 밝히기.
2. 코드는 최소한으로 — 혹시 몰라서 미리 넣는 기능 금지.
3. 이미 저장된 스킬을 고칠 때는 필요한 부분만 정확히 수정.
4. 성공 기준을 검증할 수 있는 테스트로 삼고, 통과할 때까지 반복.

이 원칙들은 `prompts.py`에 박혀 있습니다. LLM이 흔히 빠지는 과잉 설계 경향을 막고, 코드 생성을 작업 목적에 집중시키는 핵심 장치입니다.

## 3. 한계 및 개선 가능 방향

### 현재의 한계

**보안: 진짜 샌드박스가 아닙니다.**
`run_python`은 호스트 사용자 권한으로 돌아갑니다. 믿을 수 없는 입력을 다루려면 서브프로세스를 컨테이너(firecracker, nsjail, Docker exec)로 바꿔야 합니다.

**히스토리 관리가 거칩니다.**
히스토리 자르기가 메시지 수 기준이지 토큰 기준이 아닙니다. 도구 결과 하나가 거대한 파일 덤프라면 20개 메시지 안에서도 컨텍스트 창을 넘길 수 있습니다.

**환각 방지가 최선 노력 수준입니다.**
`ask_user`에 빈 입력이 들어왔을 때 보내는 "no input" 신호는 모델이 무시할 수도 있는 문자열일 뿐입니다. 더 강한 방어는 `run_turn()` 안에 넣는 형태가 됩니다 — assistant가 NO-INPUT 도구 결과 직후에 최종 답변을 내려고 하면 거부하고 `ask_user`를 다시 호출하도록 강제.

**시스템 프롬프트가 긴 서두를 유도합니다.**
"THINK BEFORE CODING"이 매 턴 여러 단락의 사전 설명을 만들어내며 TPM 예산을 잡아먹습니다.

**그 밖의 한계:**
`load_skill`은 호출 가능한 객체가 아니라 소스 텍스트를 돌려줍니다. 스트리밍이 없어서 매 응답이 다 올 때까지 기다려야 하고, 종료하면 세션 상태가 사라지며, 스킬은 Python만 지원합니다. 자동 테스트도 아직 없습니다.

### 해결한 문제 (2026-05-05)

초기 Groq 연동 과정에서 드러난 버그들을 첫 번째 사용성 패치에서 처리했습니다. 겉으로는 모델 문제처럼 보였지만 실제로는 에이전트 쪽 버그였습니다.

| 증상 | 실제 원인 | 수정 |
|---|---|---|
| 사용자가 JSON을 붙여넣었는데도 에이전트가 계속 JSON을 달라고 함 | `input()`은 한 줄만 읽음; 나머지 줄이 stdin 버퍼에 남아서 다음 `ask_user`에 빈 답변으로 들어감 | `_read_user_input()`이 `select`로 버퍼를 비워서 여러 줄 붙여넣기가 하나의 메시지로 도달하게 함 |
| `ask_user`가 빈 입력을 받으면 에이전트가 답을 지어냄(환각) | `tools.py`가 애매한 `"(user gave no answer)"` 문자열을 반환 | `"USER PROVIDED NO INPUT — do not guess or fabricate"` 라는 명확한 신호로 교체 |
| `429 (TPM)` 오류가 세션을 죽임 | `call_llm`이 모든 `HTTPError`에 대해 `sys.exit` 호출 | 최대 2회 자동 재시도; Groq 응답 본문의 대기 시간 파싱, 30초 상한 적용 |
| 무료 티어 TPM이 몇 턴 만에 바닥남 | 매 호출마다 전체 메시지 히스토리를 보내고, 끝없이 늘어남 | `_trim_history`가 히스토리를 20개 메시지로 제한; tool-result 메시지가 홀로 떨어지지 않도록 user-message 경계에서 잘라냄 |

### 개선 단계 (계획)

새 작업을 시작할 때 우선순위 판단 기준으로 쓰세요.

**Tier 1 — 꼭 필요한 사용성 수정.** 위 4개 수정으로 완료.

**Tier 2 — 안정성 & 안전성.**
토큰 기반 히스토리 압축 (자르기로 부족하면 요약으로 대체), `run_python`을 위한 진짜 샌드박스, 모델 대체 체인 (429가 계속 나면 자동으로 다른 모델로 전환), `run_turn` 자체에서의 더 강한 환각 방지.

**Tier 3 — 품질 다듬기.**
시스템 프롬프트 줄이기, 스트리밍 응답, 디스패치 계층 위의 pytest 스위트, 세션 간 대화 저장, 다른 언어 스킬 지원 (shell/Node).

## 부록: 검증한 시나리오 (2026-05-05)

`openai/gpt-oss-20b` (Groq 무료 티어, `GROQ_MAX_TOKENS=1024`)에서 처음부터 끝까지 돌려봤습니다. 7개 시나리오 모두 통과.

| # | 시나리오 | 검증 포인트 |
|---|---|---|
| 1 | 인라인 JSON과 여러 줄 붙여넣기 | JSON이 쪼개지지 않고 하나의 메시지로 도달하는지 |
| 2 | 모호한 요청 ("파일 정리 좀 해줘") | 추측 없이 `ask_user`로 먼저 확인하는지 |
| 3 | 간단한 계산 (7의 배수 + 자릿수 합 조건) | 짧은 스크립트 한 번으로 끝내는지, 과잉 설계하지 않는지 |
| 4 | 스킬 만들기 ("앞으로 자주 쓸 거야") | `save_skill` → y/n 확인 → `skills/` 저장 흐름 |
| 5 | 여러 단계 + 파일로 상태 전달 (CSV 생성 → 읽기 → 분석) | 한 턴 안에서 `run_python` 여러 번, workspace 파일 경유 |
| 6 | 도구가 필요 없는 지식 질문 (게임 기획) | 도구 호출 없이 바로 답변하는 자제력 |
| 7 | 여러 턴 창작 코딩 (기획 → pygame 프로토타입 → 실행 안내) | 턴 사이 모드 전환, headless로 못 돌리는 코드는 코드 블록으로 출력하는 판단 |

스킬 재사용도 매끄러웠습니다 — 한 턴에서 저장한 스킬이 이후 턴에서 `load_skill`로 바로 쓰였고, 세션을 재시작해도 사용자가 경로나 스킬 이름을 몰라도 자연어만으로 찾아 쓸 수 있었습니다. 이게 되는 이유는 시작할 때 각 스킬의 `SKILL.md` 설명만 시스템 프롬프트에 넣고, 모델이 사용자 의도와 의미적으로 대조해서 필요한 스킬을 골라 불러오기 때문입니다.

세션 안의 메시지 히스토리는 의도적으로 세션 사이에 유지되지 않습니다 — 그건 스킬이 할 일입니다.
