# Agentic Shell

자연어로 받은 요청을 Python 도구를 그때그때 만들어 실행하며 처리하는 작은 CLI 에이전트입니다. 만든 도구가 나중에도 쓸모가 있겠다 싶으면, 사용자에게 확인을 받고 로컬 `skills/` 디렉토리에 저장해 둡니다. 이후 세션에서 에이전트가 알아서 찾아 씁니다.

전체 제어 흐름은 표준 라이브러리 Python만으로 직접 짰습니다 — 에이전트 라이브러리도, SDK도 쓰지 않고, Groq의 OpenAI 호환 Chat Completions API를 `urllib`로 호출하는 게 전부입니다.

## 빠른 시작

```bash
# 필요한 것: Python 3.10+, Groq API 키 (무료 티어면 충분)
export GROQ_API_KEY=gsk_...

# (선택) 아래 조합을 시작점으로 추천합니다
export GROQ_MODEL="openai/gpt-oss-20b"   # 무료 티어에서 tool calling이 안정적
export GROQ_MAX_TOKENS=1024              # 무료 티어 TPM 한도 부담을 줄여 줌

python3 agent.py
```

여러 줄 붙여넣기를 지원합니다 — 프롬프트와 데이터를 한꺼번에 붙여넣으면 하나의 메시지로 처리됩니다:

```
you> 다음 JSON에서 HP가 100보다 큰 몬스터 이름을 뽑고, 그들의 평균 HP를 알려줘.
[
  {"name": "Slime",  "hp":  30},
  {"name": "Orc",    "hp": 120},
  {"name": "Dragon", "hp": 800},
  {"name": "Lich",   "hp": 250}
]

agent> Plan: parse the JSON, filter hp > 100, average those. Criterion: print
       qualifying names and their mean hp.

[tool: run_python]
agent> Orc, Dragon, Lich — 평균 HP 390.0.
       이 로직을 'monster_hp_summary' 스킬로 저장해둘까요?

you> y
[saves to skills/monster_hp_summary/]
```

## 프로젝트 구조

```
agent.py        메인 루프 — API를 돌리는 while 루프 하나
tools.py        내장 도구 스키마 + 디스패치
prompts.py      4가지 운영 원칙이 담긴 시스템 프롬프트
skills/         사용자 확인을 거쳐 저장된 도구들 (스킬마다 SKILL.md + tool.py)
workspace/      run_python이 쓰는 임시 디렉토리
```

### 내장 도구

| 도구          | 하는 일                                                       |
| ------------- | ------------------------------------------------------------- |
| `run_python`  | 격리된 서브프로세스에서 Python 스크립트 실행                  |
| `read_file`   | UTF-8 텍스트 파일 읽기                                        |
| `write_file`  | workspace에 파일 쓰기                                         |
| `ask_user`    | 확인이 필요할 때 사용자에게 질문 (HITL)                       |
| `save_skill`  | 만든 도구 저장하기 — 반드시 y/n 확인을 거침                   |
| `load_skill`  | 저장된 스킬의 소스 코드 불러오기                              |
| `list_skills` | 저장된 스킬 목록과 "언제 쓰면 좋은지" 설명 보기              |

## 설계 결정

**SDK도, 프레임워크도 없이 직접 HTTP 호출.**
`agent.py`는 `urllib`로 `https://api.groq.com/openai/v1/chat/completions`에 직접 POST합니다. 제어 흐름은 `run_turn()` 안의 `while` 루프 하나가 모델을 움직이는 구조입니다: 메시지를 보내고 → `tool_calls`를 받고 → 도구를 실행하고 → `tool` role 결과를 붙이고 → 다시 반복. 의도적인 선택입니다 — 에이전트 추상화 계층을 쓰면 이 연습에서 직접 다뤄 보고 싶은 부분이 가려지니까요.

**서브프로세스로 격리 실행.**
`run_python`은 모델이 작성한 코드를 `workspace/_run.py`에 쓰고, 새 `python3` 서브프로세스에서 타임아웃을 걸어 실행합니다. 가장 단순하면서도 돌아가는 격리 방식입니다 — 호출 사이에 프로세스 상태가 섞이지 않고, stdout/stderr가 깔끔하게 분리되며, 모델은 실제 traceback을 `tool_result`로 받아볼 수 있습니다. 이 덕분에 스스로 오류를 고치는 게 가능합니다.

**스킬은 코드가 아니라 데이터로 취급.**
저장된 스킬은 파일 두 개짜리 디렉토리입니다:
- `SKILL.md` — "언제 쓰면 좋은지" 한 단락 설명
- `tool.py` — 실제 코드

시작할 때 에이전트는 모든 `SKILL.md`를 읽되 *설명만* 시스템 프롬프트에 넣습니다(소스는 넣지 않음). 전체 소스는 에이전트가 그 스킬을 쓰겠다고 판단했을 때 `load_skill`로 불러옵니다. 스킬이 많아져도 프롬프트가 작게 유지되는 구조입니다 — 메뉴를 보고, 필요한 레시피만 꺼내 쓰는 셈이죠.

**명시적으로 사람이 개입하는 접점 3곳.**
1. 요청이 모호하거나 빠진 정보가 있으면 `ask_user`로 물어봄.
2. `save_skill`은 항상 미리보기를 보여주고 y/n 확인을 받음.
3. 사용자가 매 작업을 `you>` 프롬프트에서 직접 입력.

시스템 프롬프트에서 모델에게 "추측하지 말고 `ask_user`로 확인하라"고 명시하고 있습니다.

**Traceback 피드백으로 스스로 고치기.**
사용자 턴 하나당 모델이 최대 12번까지 반복할 수 있습니다. `run_python`이 0이 아닌 종료 코드를 내면, 전체 STDOUT/STDERR/종료코드가 다음 `tool_result`로 돌아갑니다. 시스템 프롬프트는 모델에게 traceback을 잘 읽고 *진짜 원인*을 고쳐서 다시 돌리라고 지시합니다 — 대충 때려맞추지 말고요.

**시스템 프롬프트에 담긴 4가지 운영 원칙.**
모델은 다음을 따르도록 지시받습니다:
1. 코드를 작성하기 전에 가정과 성공 기준을 먼저 밝히기.
2. 코드는 최소한으로 — 혹시 몰라서 미리 넣는 기능 금지.
3. 이미 저장된 스킬을 고칠 때는 필요한 부분만 정확히 수정.
4. 성공 기준을 검증할 수 있는 테스트로 삼고, 통과할 때까지 반복.

이 원칙들은 `prompts.py`에 박혀 있습니다. LLM이 흔히 빠지는 과잉 설계 경향을 막고, 코드 생성을 작업 목적에 집중시키는 핵심 장치입니다.

## 검증한 시나리오 (2026-05-05)

**`openai/gpt-oss-20b`** (Groq 무료 티어, `GROQ_MAX_TOKENS=1024`)에서 처음부터 끝까지 돌려봤습니다. 7개 시나리오 모두 통과.

**1. 회귀 테스트 — 인라인 JSON과 여러 줄 붙여넣기**

```
다음 JSON에서 HP가 100보다 큰 몬스터 이름을 뽑고, 그들의 평균 HP를 알려줘.
[
  {"name": "Slime",  "hp":  30},
  {"name": "Orc",    "hp": 120},
  {"name": "Dragon", "hp": 800},
  {"name": "Goblin", "hp":  45},
  {"name": "Lich",   "hp": 250}
]
```
여러 줄 붙여넣기 수정이 제대로 됐는지 확인합니다 — JSON이 여러 `input()` 호출로 쪼개지지 않고 프롬프트와 함께 하나의 메시지로 들어오는지 봅니다.

**2. 모호한 요청 → `ask_user`**

```
파일 정리 좀 해줘
```
에이전트가 추측하거나 코드부터 쓰지 않고, `ask_user`를 통해 확인해야 합니다 (어떤 파일인지, 어디 있는지, 어떤 식으로 정리할 건지).

**3. 간단한 계산 → `run_python`**

```
1부터 1000까지 중에 7의 배수이면서 자릿수의 합이 짝수인 수의 개수를 세줘.
```
짧은 스크립트 한 번이면 끝. 기본 `run_python` 흐름과, 모델이 일회성 작업을 쓸데없이 복잡하게 만들지 않는다는 걸 확인합니다.

**4. 스킬 만들기 → `save_skill`**

```
"2026-05-05" 같은 ISO 날짜를 받아서 그 날이 그 해의 몇 번째 날인지 알려주는 기능 만들어줘. 앞으로 다른 날짜로도 자주 쓸 거야.
```
"앞으로 자주 쓸 거"라는 말이 `save_skill` 제안을 끌어냅니다 (y/n으로 확인). 확인 과정과 `skills/day_of_year/` 아래 저장이 제대로 되는지 확인합니다.

**5. 여러 단계 + 파일로 상태 전달**

```
workspace에 sample.csv 만들어줘 — 가짜 학생 10명 데이터(이름, 점수). 그다음 그 파일 읽어서 평균 점수 위/아래로 나눠서 두 그룹의 이름 출력.
```
한 턴 안에서 `run_python`을 여러 번 호출하고, workspace 파일을 통해 상태를 넘기는 걸 확인합니다 (각 `run_python`은 새 서브프로세스라서 메모리는 공유되지 않음).

**6. 열린 기획 / 도구가 필요 없는 질문**

```
카피바라가 배달하는 게임 시나리오 초안을 어떻게 설계하면 좋을지 알려줘.
```
계산할 데이터도, 건드릴 파일도 없는 순수 지식 질문입니다. 에이전트는 `run_python`이나 다른 도구를 쓰지 않고 plain markdown으로 답변했습니다(8단계 게임 디자인 방법론 표). 시스템 프롬프트가 `run_python`을 주된 수단으로 제시하지만, 모델은 "그냥 대답하기"가 맞는 상황을 알아서 판단해 도구 호출을 건너뜁니다 — 이 자제력을 확인하는 시나리오입니다. 테스트 #2와 짝을 이뤄, 에이전트가 상황에 맞는 모드(물어보기 / 코딩 / 바로 대답)를 고른다는 점을 보여줍니다.

**7. 여러 턴에 걸친 창작 코딩 (기획 → 프로토타입 → 실행 안내)**

테스트 #6에서 이어진 두 턴이 같은 대화를 확장합니다:

```
you> 실제로 만들어볼래? python으로?
agent> [완성된 pygame 프로토타입을 — "Copy-Bara Delivery" — Python 코드 블록으로
       내놓음: 키보드로 조작하는 카피바라, 충돌 감지 포함 아이템 픽업,
       배달 지점, 전체 게임 루프]

you> 어떻게 실행해?
agent> [번호 매긴 실행 안내: Python 버전 확인, pip install pygame, .py 파일 만들기,
       실행, 키 바인딩, 문제 해결 팁]
```

두 가지를 확인합니다:

- **턴 사이의 모드 전환.** 같은 에이전트가 자문형 글(#6) → 실제 돌아가는 프로토타입 → 단계별 실행 안내로 자연스럽게 넘어갑니다. 짧은 한마디면 충분하고, 프롬프트를 다시 잡거나 맥락을 초기화할 필요가 없습니다.
- **headless로 돌릴 수 없는 코드에 대한 판단.** pygame 앱은 화면이 필요하기 때문에, 에이전트가 `run_python`(디스플레이 없는 격리 서브프로세스)으로 실행하려 들지 않고 채팅에 코드 블록으로 보여줬습니다. 올바른 판단입니다 — `run_python`은 stdout/stderr로 결과를 확인할 수 있는 코드에 쓰는 것이지, GUI나 인터랙티브 프로그램에 쓰는 게 아닙니다.

### 돌려보면서 알게 된 것

- **스킬 재사용이 매끄러웠습니다.** 한 턴에서 저장한 스킬(예: #4의 `day_of_year`)이 다음 턴 시스템 프롬프트의 `Saved skills` 목록에 나타났고, 모델은 새로 만들지 않고 `load_skill`로 가져다 썼습니다.
- **세션을 다시 시작해도 재사용이 잘 됩니다 — 스킬 이름이나 경로를 몰라도 됩니다.**
  에이전트를 재시작한 뒤, 사용자가 경로도 파일명도 스킬 이름도 전혀 알려주지 않고 자연어로 모호하게 말했는데도 `day_of_year`를 찾아 썼습니다:

  ```
  you> 내가 아까 ISO 날짜를 받아서 그 날이 그 해의 몇 번째 날인지 알려주는
       기능을 만들었는데 한번 사용해볼 수 있을까?

  [tool: ask_user]
  agent asks> Please provide an ISO date string (YYYY-MM-DD) to test the
              day_of_year tool.
  you> 1994-09-19

  [tool: load_skill]
  [tool: run_python]
  agent> The ISO date **1994-09-19** is the **262nd** day of the year.
  ```

  이게 되는 이유는 `load_skills_index()`가 시작할 때 각 스킬의 `SKILL.md`를 읽어 *"언제 쓰면 좋은지"* 설명만 시스템 프롬프트에 넣기 때문입니다. 모델이 사용자의 의도를 그 설명과 의미적으로 대조한 뒤 `load_skill(name=...)`로 소스를 가져옵니다. 사용자는 스킬이 디스크 어디에 있는지 신경 쓸 필요가 없습니다.
- 세션 안의 메시지 히스토리는 *의도적으로* 세션 사이에 유지되지 않습니다 — 그건 스킬이 할 일입니다. (대화 영속화는 아래 Tier-3 미해결 항목으로 남아 있습니다.)

## 알려진 이슈 & 로드맵

이 섹션은 발견하고 고친 것, 앞으로 고쳐야 할 것을 기록하는 살아있는 변경 이력입니다. 새로 발견한 내용을 같은 형식으로 추가하면, 다음 세션이 이전 세션이 멈춘 곳에서 바로 이어 작업할 수 있습니다.

### 최근 해결 (2026-05-05)

초기 Groq 연동 과정에서 드러난 버그들을 첫 번째 사용성 패치에서 처리했습니다. 겉으로는 모델 문제처럼 보였지만 실제로는 에이전트 쪽 버그였습니다.

| 증상 | 실제 원인 | 수정 |
|---|---|---|
| 사용자가 JSON을 붙여넣었는데도 에이전트가 계속 JSON을 달라고 함 | `input()`은 한 줄만 읽음; 나머지 줄이 stdin 버퍼에 남아서 다음 `ask_user`에 빈 답변으로 들어감 | `agent.py`의 `_read_user_input()`이 `select`로 버퍼를 비워서 여러 줄 붙여넣기가 하나의 메시지로 도달하게 함 |
| `ask_user`가 빈 입력을 받으면 에이전트가 답을 지어냄(환각) | `tools.py`가 애매한 `"(user gave no answer)"` 문자열을 반환했음 | `"USER PROVIDED NO INPUT — do not guess or fabricate"` 라는 명확한 신호로 교체 |
| `429 (TPM)` 오류가 세션을 죽임 | `call_llm`이 모든 `HTTPError`에 대해 `sys.exit` 호출 | 최대 2회 자동 재시도; Groq 응답 본문의 `try again in Xs`에서 대기 시간을 파싱하되 30초 상한 적용 |
| 무료 티어 TPM이 몇 턴 만에 바닥남 | 매 호출마다 전체 메시지 히스토리를 보내고, 끝없이 늘어남 | `_trim_history`가 히스토리를 20개 메시지로 제한 (env: `GROQ_HISTORY_CAP`); tool-result 메시지가 홀로 떨어지지 않도록 user-message 경계에서 잘라냄 |

### 모델별 주의사항

- **Groq에서 모델마다 tool-calling 안정성이 다릅니다.**
  `llama-3.3-70b-versatile`(원래 기본값)은 가끔 정상적인 `tool_calls` 필드 대신 Llama 고유의 `<function=name{json}</function>` 평문 형식으로 도구를 호출합니다. 이러면 Groq 서버 파서가 `tool_use_failed`로 400 에러를 냅니다. 무료 티어에서는 `openai/gpt-oss-20b`를 권장합니다.
- **작은 모델은 답이 아닙니다.** `llama-3.1-8b-instant`는 TPM 한도가 더 넉넉하지만 에이전트 루프를 돌리기엔 역부족입니다 — 인라인 데이터를 무시하고, 같은 질문을 반복하며, 진전 없이 토큰만 소비합니다. 간단한 일회성 작업에만 쓰세요.
- **무료 티어에서는 TPM이 가장 큰 제약입니다.** 매 호출이 전체 히스토리 + 도구 스키마 + 시스템 프롬프트를 보내고, 응답용으로 `GROQ_MAX_TOKENS=4096`을 예약하면 단일 요청이 분당 8k 예산의 5k 이상을 잡아먹을 수 있습니다. `GROQ_MAX_TOKENS`를 ~1024로 낮추는 게 가장 쉬운 해결책입니다.

### 미해결

- **진짜 샌드박스가 아닙니다.** `run_python`은 호스트 사용자 권한으로 돌아갑니다. 믿을 수 없는 입력을 다루려면 서브프로세스를 컨테이너(firecracker, nsjail, Docker exec)로 바꿔야 합니다.
- **히스토리 자르기가 메시지 수 기준이지, 토큰 기준이 아닙니다.** 도구 결과 하나가 거대한 파일 덤프라면 20개 메시지 안에서도 컨텍스트 창을 넘길 수 있습니다. 토큰 추정기 + 요약 패스가 다음 단계입니다.
- **환각 방지가 최선 노력 수준입니다.** "no input" 신호는 모델이 무시할 수도 있는 문자열일 뿐입니다. 더 강한 방어는 `run_turn()` 안에 넣는 형태가 됩니다 — assistant가 NO-INPUT 도구 결과 직후에 최종 답변을 내려고 하면 거부하고 `ask_user`를 다시 호출하도록 강제.
- **시스템 프롬프트가 긴 서두를 유도합니다.** "THINK BEFORE CODING"이 매 턴 여러 단락의 사전 설명을 만들어내며 TPM 예산을 잡아먹습니다. "가정과 기준을 한 문장으로, 그 다음 바로 행동"으로 프롬프트를 조이는 건 Tier-3 후속 작업입니다.
- **스트리밍 없음.** 매 응답이 다 올 때까지 기다려야 합니다.
- **대화 저장 없음.** 종료하면 세션 상태가 사라집니다.
- **`load_skill`은 호출 가능한 객체가 아니라 소스 텍스트를 돌려줍니다.** 점진적 노출을 위한 절충안입니다 — 스킬마다 도구를 등록하면 더 깔끔하겠지만 프롬프트 토큰 비용이 올라갑니다.
- **Python 전용 스킬.** shell이나 Node에서도 같은 패턴이 통하겠지만, 아직 연결해 놓지 않았습니다.
- **자동 테스트 없음.** `call_llm`을 mock하고 디스패치 계층을 검증하는 pytest 스위트를 만들면 리팩토링이 훨씬 안전해질 것입니다.

### 개선 단계 (계획)

새 작업을 시작할 때 우선순위 판단 기준으로 쓰세요:

- **Tier 1 — 꼭 필요한 사용성 수정.** *위 4개 수정으로 완료.*
- **Tier 2 — 안정성 & 안전성.**
  - 토큰 기반 히스토리 압축 (자르기로 부족하면 요약으로 대체)
  - `run_python`을 위한 진짜 샌드박스
  - 모델 대체 체인 (429가 계속 나면 자동으로 다른 모델로 전환)
  - `run_turn` 자체에서의 더 강한 환각 방지
- **Tier 3 — 품질 다듬기.**
  - 시스템 프롬프트 줄이기 (턴당 서두 축소)
  - 스트리밍 응답
  - 디스패치 계층 위의 pytest 스위트
  - 세션 간 대화 저장
  - 다른 언어 스킬 지원 (shell/Node)
